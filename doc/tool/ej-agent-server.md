Поддерживается начиная с версии [3.14.0](../change/изменения_в_версии_3.14.0.md).

Компонент ej-agent-server предназначен для
[распределённой компиляции](../параллельная_компиляция.md) и
[распределённого тестирования](../параллельное_тестирование.md).
Он поддерживает протокол обмена, реализованный в [ej-agent](../tool/ej-agent.md),
но вместо доступа по протоколу ssh клиент взаимодействует с сервером по
протоколу websocket. Это позволяет реализовывать распределённую компиляцию и тестирование
в конфигурации, когда к серверу ejudge доступ возможен только по протоколам http/https.
Кроме того, использование протокола websocket вместо ssh более безопасно, так как у клиента
нет возможности воспользоваться ssh-доступом для других целей.

### Компиляция

Компонент ej-agent-server реализован поверх библиотеки [libwebsockets](https://libwebsockets.org),
которая должна быть скомпилирована с поддержкой [libuv](https://libuv.org).

Библиотеку libuv можно скомпилировать самостоятельно, либо использовать версию, поставляемую
с дистрибутивом linux. Например, чтобы установить libuv на Fedora Linux потребуется команда

```bash
sudo dnf install libuv-devel
```

В дистрибутиве Fedora 40 поставляется версия 1.51, являющаяся текущей на момент написания данной
документации.

Библиотеку libwebsockets потребуется собрать самостоятельно, так как нужна статическая сборка
с поддержкой libuv. Для этого склонируйте исходный код с github:

```bash
git clone -b v4.5-stable --single-branch https://github.com/warmcat/libwebsockets.git
```

Создайте и перейдите в каталог сборки:

```bash
mkdir libwebsockets/build
cd libwebsockets/build
```

Запустите cmake для конфигурирования:

```bash
cmake -DCMAKE_INSTALL_PREFIX:PATH=/home/cher/lws -DLWS_WITH_SHARED=OFF -DLWS_WITH_LIBUV=ON ..
```

Скомпилируйте и проинсталлируйте библиотеку:

```bash
make
sudo make install
```

Чтобы ejudge при компиляции нашёл библиотеку libwebsockets и смог её использовать
потребуется запуск configure с опциями `--with-libwebsockets-include` и `--with-libwebsockets-lib`.
Для этого перейдите в каталог с исходным кодом ejudge и перезапустите в нем скрипт `./configure`,
добавив эти опции, например:

```bash
./configure OPTIONS --with-libwebsockets-include=/opt/lws/include --with-libwebsockets-lib=/opt/lws/lib64
```

Здесь `OPTIONS` &mdash; это опции, с которыми был сконфигурирован ejudge до этого.
Их можно посмотреть в файле `config.log` в его начале. Если библиотеки libuv и libwebsockets
найдены и их можно использовать, в заголовочном файле `config.h` будут определены
макросы `CONF_HAS_LIBUV` и `CONF_HAS_LIBWEBSOCKETS`:

```c
/* libuv available */
#define CONF_HAS_LIBUV 1

/* libwebsockets available */
#define CONF_HAS_LIBWEBSOCKETS 1
```

### Конфигурирование на стороне сервера ejudge

По умолчанию компонент ej-agent-server не стартует при запуске ejudge. В файл [`ejudge.xml`](../ejudge.xml.md) необходимо добавить параметры запуска на стороне сервера. Минимальная конфигурация
на стороне сервера следующая:

```xml
  <agent_server enable="true">
    <token_file>agent_passwd</token_file>
  </agent_server>
```

Элемент `agent_server` должен находиться на верхнем уровне конфигурации, то есть
в элементе `config`. Элемент `token_file` задаёт имя файла с токеном доступа, который будет проверяться при каждом подключении к ej-agent-server. В данном примере файл
называется `agent_passwd` и находится в том же каталоге, в котором находится файл ejudge.xml.
Файл `agent_passwd` должен позволять доступ на чтение только пользователю ejudge. Это можно
сделать следующими командами:

```bash
chown ejudge:ejudge agent_passwd
chmod 600 agent_passwd
```

Файл должен содержать одну непустую строку: токен доступа к серверу. Например, файл может
иметь такое содержимое:

```
32N63wHFAXd9m9Sqm7rrRgZ4vxTE3K7z
```

Внимание, это не настоящий токен! Не копируйте этот токен в свои конфигурационные файлы!

Это означает, что для доступа к серверу клиент должен будет передать HTTP-заголовок
следующего вида:

```
Authorization: Bearer 32N63wHFAXd9m9Sqm7rrRgZ4vxTE3K7z
```

Если ej-agent-server был скомпилирован и корректно сконфигурирован, при старте
системы ejudge с помощью ejudge-control будет выведена информация о ej-agent-server,
например:

```bash
$ ejudge-control start
2026-01-18T23:22:11Z:info:ej-users 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-super-server 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-compile 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-super-run 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-jobs 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-agent-server 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
2026-01-18T23:22:11Z:info:ej-contests 3.13.0+ (GIT d93e68363), compiled 2026-01-02 20:52:07
```

По умолчанию ej-agent-server ожидает подключений на порту 9204. Сконфигурируйте
nginx или apache, чтобы запросы к некоторому URL проксировались на порт 9204.
Например, если используется nginx, для проксирования запросов к `wss://SERVER/ej-agent-server`
добавьте в соответствующий конфигурационный файл nginx следующий фрагмент:

```
    location /ej-agent-server {
        proxy_pass http://127.0.0.1:9204/ej-agent-server;
        proxy_http_version      1.1;
        proxy_set_header        Upgrade $http_upgrade;
        proxy_set_header        Connection "Upgrade";
        proxy_set_header        Host            $http_host;
        proxy_set_header        X-Real-IP       $remote_addr;
        proxy_set_header        X-Forwarded-For $proxy_add_x_forwarded_for;
    }
```

После рестарта nginx URL должен быть доступен.
Проверить подключение можно с помощью утилиты curl:

```bash
curl -v -H 'Authorization: Bearer 32N63wHFAXd9m9Sqm7rrRgZ4vxTE3K7z' 'wss://SERVER/ej-agent-server?queue_id=test&mode=run'
```

При успехе в конце вывода будет примерно такое, после чего curl будет ожидать ввод данных.

```
< Connection: upgrade
< Upgrade: WebSocket
< Sec-WebSocket-Accept: asfdskfskdfaskfasdf
* Received 101, switch to WebSocket; mask ffeeffee
```

Здесь `SERVER` &mdash; это имя вашего сервера.

### Запуск и остановка сервера

Если ej-agent-server скомпилирован, и если в файле конфигурации ejudge.xml
задан элемент `agent_server`, компонент ej-agent-server будет запускаться и
останавливаться вместе со всеми остальными компонентами ejudge.
Если необходимо запустить только ej-agent-server, можно выполнить команду

```bash
ejudge-control start agent-server
```

Если необходимо остановить только ej-agent-server, можно выполнить команду

```bash
ejudge-control stop agent-server
```

### Опции конфигурации сервера

Элемент `agent_server` в файле ejudge.xml может содержать следующие элементы:

```xml
  <agent_server enable="ENABLE">
    <token_file>TOKEN_FILE</token_file>
    <rules_file>RULES_FILE</rules_file>
    <port>PORT</port>
    <service>SERVICE</service>
  </agent_server>
```

* `ENABLE` &mdash; булевское значение. Если оно равно `true` (или `yes`), при запуске ejudge будет запускаться компонент ej-agent-server.
* `TOKEN_FILE` &mdash; файл с секретным токеном. Чтобы другие пользователи не имели к нему доступа, владельцем и группой этого файла должна быть ejudge, а права доступа &mdash; 600. Если имя файла относительное, путь к файлу определяется относительно каталога, в котором находится файл ejudge.xml.
* `RULES_FILE` &mdash; файл с ограничениями доступа к файловой системе сервера. Ограничения доступа проверяются, когда клиент пытается скачать с сервера какой-либо файл, например, файл с тестовыми данными. Ограничения доступа нужны, чтобы клиент не смог скачать, например, важные системные файлы. Если имя файла относительное, путь к файлу определяется относительно каталога, в котором находится файл ejudge.xml.
* `PORT` &mdash; номер порта, на котором компонент ej-agent-server ожидает подключений по протоколу websocket. По умолчанию номер порта равен 9204.
* `SERVICE` &mdash; имя сервиса websocket. По умолчанию имя сервиса равно `ej-agent-server`, это имя должно использоваться в URL при подключении к сервису.

При использовании nginx в качестве reverse proxy для подключения к серверу номер порта и имя
сервиса указываются в URL, задаваемом в директиве `proxy_pass`.

### Ограничения доступа к файловой системе сервера

При компиляции и тестировании помимо файла с тестируемой программой используются и другие файлы.
Например, это могут быть тестовые данные, проверяющая программа и так далее.
Агенты компиляции и тестирования ej-compile и ej-super-run поддерживают загрузку
необходимых файлов и кеширование на стороне агента. Для этого
в протоколе взаимодействия клиента и сервера поддерживается специальная команда mirror.
Чтобы агент компиляции и тестирования не мог скачать любой файл из файловой системы
сервера в ej-agent-server реализована проверка на допустимые для скачивания пути
в файловой системе.

По умолчанию для скачивания разрешёны каталоги `${EJUDGE_CONTESTS_HOME_DIR}/NNNNNN/`,
где `NNNNNN` &mdash; это номер турнира, но при этом запрещены для скачивания каталоги 
`${EJUDGE_CONTESTS_HOME_DIR}/NNNNNN/conf` и ``${EJUDGE_CONTESTS_HOME_DIR}/NNNNNN/var`.
Кроме того разрешён для скачивания каталог `${EJUDGE_SCRIPT_DIR}`.
При обычных параметрах конфигурирования ejudge это будут каталоги `/home/judges/NNNNNN`
и `/opt/ejudge/libexec/ejudge`.

Дополнительные ограничения или разрешения на скачивание файлов можно задать в файле
правил, имя которого задаётся в элементе `rules_file` конфигурации. Файл должен
иметь права на чтение и запись (600) только для пользователя ejudge.
Это &mdash; файл в формате JSON следующей структуры:

```json
{
    "rules": [
        {
            "kind": "prefix",
            "accept": true,
            "path": "/home/judges/public"
        },
        {
            "path": "/tmp/"
        },
        {
            "accept": true,
            "path": "/somepath/"
        }
    ]
}
```

Атрибут `rules` задаёт массив правил фильтрации. Каждое правило фильтрации
представляет собой объект, содержащий три атрибута:

* `kind` &mdash; тип правила. В настоящий момент поддерживаются только правила типа `prefix`. Этот тип правил является типом по умолчанию в случае, если атрибут `kind` отсутствует.
* `accept`: если установлен в `true`, файлы с путями с заданным префиксом разрешены к скачиванию. Значение по умолчанию `false`.
* `path` &mdash; префикс пути к файлу, который вызывает срабатываение данного правила.

Правила применяются в порядке их следования в массиве, при этом первое правило,
на котором путь к файлу удовлетворяет условию, определяет результат проверки.
Проверка на этом заканчивается, и дальнейшие правила не проверяются.

### Опции командной строки

Компонент ej-agent-server запускается следующей командой:

```bash
ej-agent-server [OPTIONS...] [CONFIG-PATH]
```

`CONFIG-PATH` &mdash; это путь к конфигурационному файлу ejudge.xml. Если этот аргумент не указан,
используется путь по умолчанию, заданный при конфигурировании системы ejudge перед её компиляцией.

Поддерживаются следующие опции командной строки:

| Опция | Описание |
|-------|----------|
| `--help` | Вывести краткое описание использования работы и завершиться с кодом 0 |
| `--version` | Вывести версию ejudge |
| `-D` | Запустить в фоновом режиме (режиме демона) |
| `-u USER` | Запуск компонента выполнять из-под пользователя `USER`. Эта опция необходима, когда компонент запускается пользователем root. Если опция не указана, используется пользователь, указанный при компиляции ejudge. Обычно этот пользователь &mdash; ejudge. |
| `-g GROUP` | Запуск компонента выполнять из-под группы `GROUP`. Эта опция необходима, когда компонент запускается пользователем root. Если опция не указана, используется группа, указанная при компиляции ejudge. Обычно эта группа &mdash; ejudge. |
| `-C DIR` | Установить каталог `DIR` в качестве рабочего каталога при работе программы. |

### Подключение клиента к ej-agent-server

Для поддержки подключения по протоколу websocket система ejudge должна быть скомпилирована с достаточно новой версией библиотеки [libcurl](https://curl.se). 
Полноценная поддержка протокола реализована в версии 8.11, но рекомендуется
установить последнюю версию библиотеки.
Руководство по компиляции и инсталляции последней версии библиотеки libcurl
доступно [здесь](build/компиляция_libcurl.md).

Если библиотека libcurl поддерживает протокол websocket,
это будет автоматически определено при конфигурации системы с
помощью [configure](configure.md), и поддержка подключения по протоколу websocket
будет скомпилирована автоматически.

Агенты тестирования [ej-compile](ej-compile.md) и [ej-super-run](ej-super-run.md) и оболочка запуска [ejudge-control](ejudge-control.md) поддерживают опцию `--agent`. Для использования подключения по вебсокетам необходимо указать префикс `ws:` в строке агента, например: `ws:wss://server.com/ej-agent-server`. Здесь `ws` &mdash; это тип агента, а `wss` &mdash; это часть URL для подключения.
